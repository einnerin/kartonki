# Пользовательские наборы (UGC) — план

Документ-черновик для будущей фичи «пользователь может создавать свои наборы слов». На момент написания (2026-04-23) фича **не реализована** — это заметка, чтобы при реализации не изобретать архитектуру заново.

## Что такое UGC в контексте Kartonki

Две категории наборов:

1. **Встроенные (curated)** — поставляются с приложением, зарегистрированы в `WordRegistry.kt`, проходят ручной контроль качества (все правила из `docs/claude/word-sets.md`), участвуют в PvP-коллекции и ачивках.
2. **Пользовательские (UGC)** — создаются игроком прямо в приложении, живут только на его устройстве (пока без cloud-синка), не проходят проверку качества, не участвуют в PvP/ачивках.

Игроку это даст возможность собирать свои словарные наборы: «Слова из «Игры престолов»», «Лексика из моего учебника», «Бизнес-термины для интервью».

## Зачем нужно разделение track'ов

Встроенные наборы жёстко стандартизированы — формат имени, обязательный `level/topic`, ровно 25 слов, разброс редкости ≤ 2, уникальность имён в теме. Это нужно для целостности продукта: предсказуемого UI, ачивок, PvP-баланса.

Если заставлять пользователя играть по этим правилам — он не будет создавать наборы. Тупо: «моя подборка слов из книги» не ложится в шаблон «Литература: основы 4». Нужны мягкие правила.

## Архитектурный паттерн: флаг `isUserCreated`

Не плодить отдельные сущности (`UserWordSetEntity`, `UserWordEntity`). Использовать **тот же** `WordSetEntity` / `WordEntity` с одним дополнительным полем:

```kotlin
@Entity(tableName = "word_sets")
data class WordSetEntity(
    // ... existing fields ...
    val isUserCreated: Boolean = false,  // новое поле
)
```

Все код-пути (квиз, репозитории, маппинги) работают одинаково. Различаются только:
1. **Валидаторы** (`find_real_dupes.py`, `validate_blank_ambiguity.py`, pre-commit хук) — пропускают наборы с `isUserCreated = true`.
2. **UI-секция** — UGC наборы показываются в отдельной вкладке «Мои наборы», без level-сортировки, без темы-группировки.
3. **PvP / ачивки** — пулят только `isUserCreated = false` (иначе чит: пользователь создаёт набор с редкими словами и фармит ачивки).
4. **FILL_IN_BLANK pipeline** — `generate_fill_in_blank_exclusions.py` не генерирует exclusions для UGC (это было бы ручной labelling, а никто не будет). Квиз FILL_IN_BLANK для UGC работает на чистой Phase 1 защите (далёкие `semanticGroup` — но у пользовательских слов `semanticGroup` может вообще отсутствовать, тогда используется базовый random shuffle).
5. **`WordDataVersion` / `WordLoader`** — триггерит re-upsert только встроенных сетов. UGC сеты сохраняются через отдельный поток (напр., `UserWordSetRepository.createSet`) и не затрагиваются миграциями.

## ID-пространство

Встроенные наборы используют блоки из `WordRegistry.languageIdBlocks`:
- en-ru: 1..999
- he-ru: 1001..1999

UGC наборы должны использовать отдельный блок, чтобы не было коллизий id:
- UGC: 100000..999999 (шесть цифр, с запасом)
- Распределение setId внутри блока — автоинкремент через Room.

## Правила, которые НЕ применяются к UGC

| Правило для built-in | UGC аналог |
|----------------------|------------|
| Формат имени `"topic: уровень [N]"` | Свободное имя от 1 до 50 символов |
| `level` обязательное (1-5) | Необязательное, может быть null |
| `topic` обязательное | Необязательное |
| Ровно 25 слов | От 1 до, например, 200 слов |
| Разброс редкости ≤ 2 смежных | Не проверяется (или даже `rarity` не обязателен) |
| Уникальность имени в (topic, lang, level) | Уникальность только в пределах юзера |
| Блок ID 1-1999 | Блок ID 100000+ |
| Все поля WordEntity обязательны | Минимум: `original`, `translation`. Остальное опционально |
| Однокоренные слова запрещены | Не проверяется |
| Дубли в теме запрещены | Не применимо (UGC вне тем) |

## Правила, которые ОСТАЮТСЯ применимы

- Синтаксис Kotlin/Room — данные должны корректно сохраняться/читаться.
- `languagePair` — обязателен (иначе не определить, какой язык изучается).
- PvP-изоляция — UGC не даёт карточки в коллекцию, не приносит ачивки.

## Шаги реализации (когда дойдут руки)

1. **Data layer:**
   - Добавить `isUserCreated: Boolean = false` в `WordSetEntity` и `WordEntity`.
   - Room-миграция: `ALTER TABLE word_sets ADD COLUMN isUserCreated INTEGER NOT NULL DEFAULT 0` (аналогично для `words`).
   - Маппинг в `Word.kt` domain-модели (поле `isUserCreated: Boolean = false`).
   - `UserWordSetRepository` — новый репо с методами `createSet`, `updateSet`, `deleteSet`, `addWord`, `removeWord`, `getUserSets`.

2. **UI:**
   - Экран «Мои наборы» — список пользовательских наборов с кнопкой «+».
   - Экран создания/редактирования набора — форма с именем и списком слов.
   - Экран добавления слова — минимальная форма `original` + `translation`, остальное опционально.

3. **Валидаторы:**
   - `find_real_dupes.py`, `validate_blank_ambiguity.py` и все другие в `scripts/validate/` — в начале каждой функции `check_*` добавить skip для наборов/слов с `isUserCreated = true`. ИЛИ фильтровать `all_sets` / `words` на входе.
   - Pre-commit хук применяется только к `WordData*.kt`, которые содержат только built-in данные — UGC живёт в БД, не в .kt файлах. Так что хук естественно не затрагивает UGC. Проверить, что это сохранится.

4. **Quiz engine:**
   - `QuizBuilder` уже работает с любым `Word` — ничего менять не нужно.
   - `pickDistractors` — когда target это UGC-слово, distractorPool может содержать как UGC, так и built-in слова. Стратегия: **не смешивать** — UGC-сессия использует только UGC-слова. Для FILL_IN_BLANK это ок: в пуле будет меньше слов, но ≥ 4 достаточно.
   - Проверка: если UGC-набор содержит < 4 слов, FILL_IN_BLANK и DEFINITION квизы недоступны (нужны 3 дистрактора).

5. **PvP / ачивки:**
   - `CollectionRepository`, `AchievementManager` — добавить фильтр `isUserCreated = false` везде, где строится коллекция или проверяются условия ачивок.

6. **Cloud sync (опционально, отдельная фича):**
   - UGC наборы синхронизируются между устройствами через Firebase/backend.
   - Только после того как UGC работает локально.

## Соображения безопасности

- **Inappropriate content.** Пользователь может ввести некорректный текст в `original` или `translation`. Пока UGC только локально — это проблема только этого пользователя. При cloud-синке (если до неё дойдёт) нужен модерационный слой.
- **Mass spam.** Нельзя давать бесконечно создавать наборы. Soft limit: ~50 UGC наборов на пользователя. Или без лимита, это дело железа.

## Когда реализовывать

Не раньше чем:
1. Завершён прогон FILL_IN_BLANK pipeline по всем встроенным наборам (Фаза 2/3 для остальных ~719 сетов).
2. Релиз в Play Store прошёл (см. `docs/claude/release-plan.md`).
3. Появились реальные запросы от пользователей на эту фичу.

Иначе UGC станет отвлекающим фактором — нужно сначала довести базовый продукт до работающего состояния.
