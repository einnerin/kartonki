# План переработки иврит-наборов (he-ru)

## Цель

Все иврит-наборы (147 штук) переработать в трёхуровневую систему:
- **Основы** (COMMON + UNCOMMON, A1–B1)
- **Продвинутый** (RARE, B2)
- **Профессиональный** (EPIC + LEGENDARY, C1–C2)

В UI наборы группируются по теме: `Медицина` → Основы / Продвинутый / Профессиональный.

Дубли между разными темами — допустимы.  
Дубли внутри одной темы (между уровнями) — запрещены.

## Фаза 1: Изменения БД

### Новые поля в `WordSetEntity` / таблице `word_sets`

```sql
ALTER TABLE word_sets ADD COLUMN topic TEXT NOT NULL DEFAULT '';
ALTER TABLE TABLE word_sets ADD COLUMN level INTEGER NOT NULL DEFAULT 0;
```

- `topic` — название темы для группировки, например `"Медицина"`, `"Технологии"`
- `level` — `0` = без уровня (старые наборы до переработки), `1` = Основы, `2` = Продвинутый, `3` = Профессиональный

### Новая таблица `word_set_membership`

```sql
CREATE TABLE word_set_membership (
    wordId INTEGER NOT NULL,
    setId  INTEGER NOT NULL,
    PRIMARY KEY (wordId, setId)
);
CREATE INDEX idx_wsm_setId ON word_set_membership(setId);
```

Одно слово может принадлежать нескольким наборам.  
При старте WordLoader заполняет эту таблицу на основе `WordEntity.setId` (обратная совместимость).

### Что остаётся без изменений

- `WordEntity.setId` — остаётся, указывает на «родной» набор слова
- Формат ID слов (`setId × 100 + position`) — остаётся для совместимости
- Версия БД: с 35 → **36**

## Фаза 3: Маппинг существующих наборов по темам

### Правила переработки

1. Читать существующий набор, смотреть на `rarity` слов
2. COMMON/UNCOMMON → уровень 1 (Основы)
3. RARE → уровень 2 (Продвинутый)
4. EPIC/LEGENDARY → уровень 3 (Профессиональный)
5. Если тема уже разбита (напр. «Медицина: основы» + «Медицина: углублённо») — переименовать и дозаполнить до 25 слов
6. Если слов много — делать «Медицина: основы 1» и «Медицина: основы 2»
7. Имя набора в БД: `"Медицина: основы"`, topic = `"Медицина"`, level = `1`

## Фаза 4: Финальная проверка

После переработки каждой темы:
```bash
python find_real_dupes.py   # 0 stolen words
python audit_all.py         # проверить деривативы внутри темы
```

Дополнительная проверка: внутри одной темы (одинаковый `topic`) дублей быть не должно ни между уровнями.

## Что НЕ трогаем

- Наборы серии «Иммигрант» (Immigrant 1–16) — отдельная серия, не переосмысляется
- Пресетные колоды (DeckSeed) — не трогаем до отдельного решения
- Английские наборы (en-ru) — после успешного завершения иврита

## Порядок работы агентов

### Параллельно (Фаза 1 + 2):
- **Агент DB**: миграция БД, обновление Entity/DAO/Repository
- **Агент UI**: группировка наборов в StudyScreen

### Последовательно (Фаза 3, по темам):
Каждый агент получает список тем и чёткую инструкцию.  
Агенты по контенту НЕ трогают одни и те же файлы одновременно.  
WordRegistry обновляет один агент за раз.

---

## ⚠️ Важные ограничения

- `WordRegistry.kt` и `WordDataVersion.kt` — только один агент за раз
- Наборы в рамках одной языковой пары — не параллельно (риск дублей)
- Каждый агент обязан запускать `python find_real_dupes.py` перед коммитом
