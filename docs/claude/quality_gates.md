# Система гарантий качества — 7 уровней защиты

## Зачем

Наборы слов — **ядро продукта**, и качество данных напрямую определяет пользовательский опыт. Ручные проверки не масштабируются (набор из 25 слов имеет ~150 полей, 37 сетов = 5550 полей). Дисциплина не помогает: коммит `f353472` захватил 60+ мусорных файлов из параллельных сессий через `git add -A` и потребовал revert. Система гарантий — про то, чтобы ни одна подобная проблема не могла тихо попасть в main.

## Семь уровней защиты

Каждый уровень — самостоятельный фильтр. Ошибка проскакивает до main только если её пропустят все семь.

| # | Уровень | Что делает | Где живёт | Когда срабатывает |
|---|---------|------------|-----------|-------------------|
| 1 | **Стандарты качества** | Фиксируют правила: живой стиль, лимиты длин, опасные близнецы | `docs/claude/quality_standards_*.md` | Читаются людьми и агентами перед работой |
| 2 | **Skills читают стандарты** | `/add-words` шаг 0: чтение 3 стандартов обязательно | `.claude/skills/add-words/SKILL.md` | При запуске skill'а |
| 3 | **Валидационные скрипты** | 7 чистых проверок + aggregator + audit | `scripts/validate/` | По запросу (ручной/CI/hook) |
| 4 | **Skill+агенты обязаны валидацию** | Блокирующее правило: `validate_all.sh` до возврата 0 | SKILL.md шаг 7, text-author/metadata-filler | Перед возвратом отчёта агентом |
| 5 | **Pre-commit hook** | Локальная защита: `.py` в корне, worktrees, submodule, WordData-валидация | `.githooks/pre-commit` | Перед каждым `git commit` |
| 6 | **CI — blocking PR workflow** | Валидация только изменённых setIds. Красный = блокирует мерж | `.github/workflows/validate-words-pr.yml` | На push/PR к main при изменении WordData |
| 7 | **CI — health weekly workflow** | Полный аудит всей базы. Красный = техсбой, не блокирует PR | `.github/workflows/validate-words-health.yml` | Еженедельно + ручной запуск |

## Установка (для новых контрибьюторов)

```bash
bash scripts/install-hooks.sh
```

Устанавливает `git config core.hooksPath .githooks` + `chmod +x` всем hook'ам. Однократно после `git clone`.

## Ручная валидация

**Один setId:**
```bash
bash scripts/validate/validate_all.sh 262
```
Прогоняет все 7 валидаторов. Exit 0 = всё хорошо, exit 1 = есть блокирующие нарушения.

**Вся база:**
```bash
bash scripts/validate/audit_all_files.sh
```
Проходит по всем setIds во всех файлах. Медленно (~15 мин). Показывает пофайловый пас/фейл.

**Только один валидатор:**
```bash
bash scripts/validate/validate_fields_filled.sh 262
bash scripts/validate/validate_group_sizes.sh 262
# ... и т.п.
```

**Регрессия на фикстурах:**
```bash
bash scripts/validate/tests/run_tests.sh
```
14 проверок: 7 валидаторов × 2 фикстуры (good_set должен пройти, bad_set должен упасть).

## Что делать при провале каждого уровня

| Уровень провалился | Действие |
|--------------------|----------|
| 1 (стандарт устарел) | Обновить `docs/claude/quality_standards_*.md`, сверить с валидаторами |
| 2 (агент не знает стандарта) | Дополнить SKILL.md/agent-spec, добавить явное чтение стандарта |
| 3 (валидатор сам сломан) | Прогнать `scripts/validate/tests/run_tests.sh`, починить до 14/14 |
| 4 (агент не прогнал валидацию) | Посмотреть его последний отчёт, прогнать вручную, исправить |
| 5 (pre-commit блокирует) | **Не использовать `--no-verify`.** Прочитать отчёт хука, исправить, повторить |
| 6 (CI PR упал) | Прогнать локально `validate_all.sh <setId>` на изменённых, исправить, force-push-free push |
| 7 (health пестрит техдолгом) | Это ожидаемо при legacy — см. `docs/claude/polishing_plan.md` раздел «Техдолг» |

## Как расширить систему — новая проверка за 3 шага

1. **Python-модуль:** `scripts/validate/validate_<check>.py` — читает `from _parser import ...`, возвращает exit 0/1.
2. **Bash-обёртка:** `scripts/validate/validate_<check>.sh` — 4 строки.
3. **Добавить в `validate_all.sh`:** одна строка в массив `CHECKS`.

Дополнительно: добавить проверку в `bad_fixture.kt` (нарушение) и в `run_tests.sh` (ожидание что validator вернёт 1). **Все контуры** (SKILL.md шаг 7, агенты, pre-commit, CI) автоматически начнут прогонять новую проверку — изменения в них не нужны, потому что все они вызывают `validate_all.sh` как чёрный ящик.

## Обход защиты

**Единственный легитимный обход** — `LARGE_COMMIT_OK=1 git commit -m "..."` для коммитов 20+ файлов, когда это сознательно (например, перенос директории или массовая нормализация). Обход **только** для warning о размере, **не** для блокирующих ошибок валидации данных.

**`git commit --no-verify` не использовать никогда.** Если хук блокирует по ошибке — исправить правило или валидатор, а не обходить. Запрещённые обходы оставляют системы с дырами.

## Инциденты, ставшие причиной системы

- **f353472** (2026-04-22) — `git add -A` захватил 60+ untracked Python-скриптов + `.claude/worktrees/agent-a054b794` как embedded git repo (submodule mode 160000). Revert, повторный чистый коммит. Уроки: запрет на `git add -A`, правила cleanliness в pre-commit hook.
- **Batch7-13 пустые** (2026-04-21) — 650 слов с `definition=null`/`example=null` несколько сессий лежали в main. text-author добавлен как инструмент, `validate_fields_filled` — как страховка.
- **adj vs adjective** (2026-04-22) — 39 слов с короткой формой `pos` ломали `QuizBuilder.pickDistractors` tier1 strict-equality. Нормализация + `validate_pos_values` + стандарт метаданных с явным запретом коротких форм.
