# Стандарт качества метаданных в Kartonki

Этот документ описывает, как должны выглядеть поля `pos` (часть речи) и `semanticGroup` (тематическая подгруппа) в `WordEntity`. Метаданные критически влияют на качество квиз-дистракторов, даже если их никто напрямую не читает на экране.

Документ — часть единой системы стандартов качества:
- [Определения](quality_standards_definitions.md) — поле `definition`/`definitionNative`
- [Примеры](quality_standards_examples.md) — поле `example`/`exampleNative`
- [Метаданные](quality_standards_metadata.md) — поле `pos`/`semanticGroup` (этот файл)

## Почему это важно

В `QuizBuilder.pickDistractors` (`app/src/main/java/com/example/kartonki/domain/quiz/QuizBuilder.kt`) дистракторы квиза ранжируются по трём уровням семантической близости:

```
sameSem = same pos AND same semanticGroup (ближайшие, правдоподобные близнецы)
diffSem = same pos only (грамматически однотипные, разная тема)
diffPos = разные pos (запасной пул)
```

Порядок различается по типу квиза (с Фазы 1 FILL_IN_BLANK pipeline, 2026-04-22):

| Тип квиза | Порядок | Цель |
|-----------|---------|------|
| TRANSLATION, DEFINITION (и все определения) | `sameSem + diffSem + diffPos` | Близкие дистракторы = сложнее квиз, игрок должен знать тонкие оттенки |
| FILL_IN_BLANK | `diffSem + sameSem + diffPos` | Далёкие дистракторы = меньше фрустрации («все варианты правильно встают в пропуск»). Близких дополнительно фильтрует поле `fillInBlankExclusions`; см. [`fill-in-blank-pipeline.md`](fill-in-blank-pipeline.md) |

Если `pos` или `semanticGroup` пустые/неконсистентные — дистракторы деградируют до случайного сэмпла, **оба** типа квизов становятся слишком лёгкими. Тот же механизм работает в `PvpGameLogic.kt` и в `WordSetRepository.getDistractorExtras()` (подтягивает кросс-сетовые дистракторы по `semanticGroup` в малых сессиях).

## Железные правила

### 1. `pos` — только длинные формы

Допустимое множество:

| Значение | Значение (рус.) | Пример слов |
|----------|-----------------|-------------|
| `noun` | существительное | apple, city, idea |
| `verb` | глагол | run, think, bake |
| `adjective` | прилагательное | red, heavy, happy |
| `adverb` | наречие | quickly, always, here |
| `number` | числительное | seven, first, twice |
| `phrase` | устойчивая многословная конструкция | thanks a lot, by the way |
| `preposition` | предлог | on, under, after |
| `interjection` | междометие | hello, wow, goodbye |
| `conjunction` | союз | and, but, because |
| `pronoun` | местоимение | who, this, my |

**Запрещённые короткие формы:** `adj`, `adv`, `interj`, `pron`. Причина: `QuizBuilder.pickDistractors` использует strict equality (`it.pos == word.pos`), и слово с `pos="adj"` не матчится со словом `pos="adjective"` даже при одинаковой semanticGroup — tier1 разваливается.

**Статус нормализации (2026-04-23):** всё чисто — 0 слов с короткими формами. Последняя партия (6 `interj` + 2 `pron` в `WordDataHebrewEveryday.kt`) выпилена sed-заменой. Валидатор `validate_pos_values.sh` блокирует регрессии.

**Как выбрать pos для неоднозначного слова:**
- Словарная форма (`original`) в словаре даётся как глагол (*to swim*) → `verb`; как существительное (*swimming* «плавание») → `noun`.
- Для составных типа *cutting board*, *transit hub* — берём головное слово: board/hub → `noun`.
- Для устойчивых фраз (*thanks a lot*, *by the way*) и меж-классовых конструкций → `phrase`.

### 2. `semanticGroup` — формат

**Snake_case**, шаблон `{topic_prefix}_{subtopic}`.

Примеры:
```
restaurant_basics, restaurant_menu, restaurant_cooking
fitness_basics, fitness_equipment, fitness_exercises
banking_operations, banking_credit
kitchen_dishware, kitchen_cutlery, kitchen_cookware
family_wedding_marriage, family_inheritance
law_id_documents, law_legal_verbs
```

**Требования:**
- Только ASCII lowercase и `_`. Без дефисов, пробелов, кириллицы.
- Префикс устоявшийся для темы — **сначала grep существующих**, только при отсутствии создавать новый.
- Для многословных тем — сокращать до читаемого: «Работа и карьера» → `job_*` или `occupation_*`; «Первая помощь» → `first_aid_*`.

### 3. Размер группы

| Размер | Статус | Комментарий |
|--------|--------|-------------|
| 1 слово | ❌ блокировка | Группа из одного слова бесполезна — нет tier1-дистракторов. Объединить с соседней |
| 2 слова | ⚠️ допустимо | Только если нет естественной соседки по оси. Допустимо для антонимических пар (loud/quiet, early/late) |
| 3–8 слов | ✅ оптимум | Дистракторы надёжно берутся из tier1 |
| 9–12 слов | ⚠️ допустимо | Если ось естественно большая (12 месяцев, 9 чисел-teens) — не дробить искусственно. Валидатор предупредит |
| 13+ слов | ❌ блокировка | Дробить на 2 подгруппы по очевидной оси |

**Исключения допустимы при явной аргументации:** months = 12 (линейный ряд, любое деление произвольно).

## Принцип «опасных близнецов»

**Главный принцип группировки:** слова в одной `semanticGroup` должны быть **правдоподобными альтернативами** друг другу в квизе. Если пользователь по простой логике отсекает дистрактор (разная ось семантики, разная роль в контексте) — группировка плохая, tier1 делает квиз слишком лёгким.

### Хорошие близнецы

| Группа | Слова | Почему работают |
|--------|-------|-----------------|
| `adj_sound` | loud, quiet | Антонимы по громкости — идеальная пара для квиза о звуке |
| `verb_senses` | look, smell, hear | Восприятие разными органами чувств; учащийся с пробелами может спутать |
| `accessory_jewelry` | ring, necklace, earring, bracelet | Украшения на разных частях тела; в def «маленький декоративный предмет на теле» все равноправны |
| `fruits_citrus` | lemon, grapefruit, tangerine | Цитрусы — одна ботаническая группа, похожий визуальный/вкусовой профиль |
| `family_wedding_roles` | bride, groom | Чистая гендерная пара на одной оси (жених/невеста) |

### Плохие близнецы

| Группа (НЕ делать) | Слова | Почему плохо |
|----|-------|--------------|
| ~~adj_physical~~ | heavy, loud, narrow | heavy (вес) и loud (звук) — разные оси, на def «тяжёлый» loud отсекается мгновенно |
| ~~adj_state~~ | early, wrong, sick, afraid | 4 разные оси в одной группе: время, корректность, здоровье, эмоция. Не близнецы |
| ~~city_mix~~ | downtown, traffic, pedestrian, landmark | Место + поток + человек + объект — не одна категория |

### Как устроить тестовую проверку

Для сомнительной группы мысленно представь MULTIPLE_CHOICE_DEFINITION квиз:
- Вопрос — одно слово из группы
- Варианты ответа — остальные 3 слова + правильный definition
- Если дистракторы отсекаются не по знанию слова, а по «очевидной разнице осей» — группа плохая

## Как подобрать префикс

**Шаг 1. Grep существующих префиксов:**

```bash
grep -hoE 'semanticGroup = "[^"]+"' app/src/main/java/com/example/kartonki/data/WordData*.kt \
  | sed 's/semanticGroup = //; s/"//g' | awk -F'_' '{print $1}' | sort -u
```

**Шаг 2.** Если тема уже встречалась — используй тот же префикс.

**Шаг 3.** Если тема новая — префикс латиницей, 4–12 символов, уникальный первый сегмент:
- ❌ `the_city_*` — артикль бесполезен
- ❌ `city_and_urban_*` — слишком длинно
- ✅ `city_*` — кратко и ясно

### Существующие префиксы (2026-04-23)

Топ-15 по частоте использования:

| Префикс | Слов | Тема |
|---------|------|------|
| `law` | 200 | Право |
| `family` | 170 | Семья |
| `food` | 168 | Еда |
| `transport` | 150 | Транспорт |
| `city` | 150 | Город |
| `weather` | 131 | Погода |
| `home` | 122 | Дом |
| `business` | 116 | Бизнес |
| `nature` | 114 | Природа |
| `work` | 100 | Работа (общее) |
| `sport` | 100 | Спорт |
| `society` | 100 | Социальное (раньше `social`) |
| `science` | 100 | Наука |
| `arch` | 100 | Архитектура (старая форма; см. примечание) |
| `advanced` | 100 | Legacy (не использовать для нового) |

**Примечание про `arch` / `architecture`:** есть оба префикса одновременно — `arch` (100 слов, старая короткая форма) и `architecture` (91 слово, новая длинная форма). Консолидация в одну конвенцию — отложенная мелкая задача. Для нового контента использовать `architecture`.

Для актуальных данных в любой момент:

```bash
grep -hoE 'semanticGroup = "[^"]+"' app/src/main/java/com/example/kartonki/data/WordData*.kt \
  | sed 's/semanticGroup = //; s/"//g' | awk -F'_' '{print $1}' | sort | uniq -c | sort -rn | head -20
```

## Самопроверка перед коммитом

- [ ] Каждое слово имеет `pos` из допустимого множества (длинная форма)
- [ ] Каждое слово имеет `semanticGroup` в snake_case
- [ ] В наборе нет групп из 1 слова (валидатор заблокирует)
- [ ] В наборе нет групп 13+ слов (валидатор заблокирует)
- [ ] Для каждой группы можно представить квиз с «опасными близнецами»
- [ ] Префикс согласован с существующей конвенцией (grep проверен)
- [ ] Валидатор `scripts/validate/validate_pos_values.sh <setId>` проходит
- [ ] Валидатор `scripts/validate/validate_group_sizes.sh <setId>` проходит

## Инфраструктура

| Агент / Skill | Где использует стандарт |
|---------------|------------------------|
| `metadata-filler` (`.claude/agents/`) | Основной исполнитель — читает этот файл перед работой |
| `add-words` (`.claude/skills/`) | Шаг 6: проставляет pos/semanticGroup при написании нового сета |
| `text-author` (`.claude/agents/`) | pos/semanticGroup вне мандата, но обязан использовать общую конвенцию |

Валидаторы в `scripts/validate/` (реализация — часть системы гарантий качества):
- `validate_pos_values.sh` — pos только из допустимого множества
- `validate_group_sizes.sh` — размеры групп в допустимых диапазонах
- `validate_all.sh` — запускает все проверки агрегированно
